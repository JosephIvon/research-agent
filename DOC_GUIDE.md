# 文档编写规范 (Documentation Style Guide)

本文档定义 AI 智能体编写项目文档的统一规范，确保文档风格一致、层次清晰、通俗易懂。

## 一、基本原则

1. **用户至上** - 假设读者是技术水平中等的用户，不是专家
2. **简洁优先** - 能用一句话说清楚的，不用三句
3. **结构清晰** - 使用标题层级、表格、列表增强可读性
4. **实例驱动** - 每个概念配一个实际例子或命令

## 二、README 结构规范

所有项目 README 必须包含以下章节，按顺序排列：

```
1. 一句话定位（项目做什么，给谁用）
2. 核心功能（表格/列表，3-8项）
3. 快速开始（3步以内完成）
4. 界面预览（可选）
5. 配置参考（精简，只列关键项）
6. 许可证
```

**禁止在 README 中包含的内容：**
- 代码修改历史记录（如"修复22项"）
- 开发过程中的讨论细节
- 内部实现细节（如"某函数第15行"）
- 尚未发布的计划功能
- 与用户使用无关的技术债务

## 三、标题层级

| 层级 | 使用场景 | 示例 |
|------|----------|------|
| H1 | 页面主标题 | `# 项目名称` |
| H2 | 章节标题 | `## 核心功能` |
| H3 | 子章节 | `### 快速开始` |
| H4 | 小节标题 | `#### Docker 部署` |

最多使用 H2，H3 用于子章节细分，H4 仅在必要时使用。

## 四、表格规范

```markdown
| 列1 | 列2 | 列3 |
|------|------|------|
| 内容 | 内容 | 内容 |
```

- 表头加粗
- 对齐一致
- 最多 4 列（超过则拆分为多个表）

## 五、代码块规范

```markdown
// 语言标记必须标注
```bash
# 终端命令
```

```python
# Python 代码
```

```yaml
# 配置文件
```
```

- 单行命令用行内代码
- 多行配置用代码块
- 敏感信息用占位符（如 `your-api-key`）

## 六、徽章 (Badges)

只在 README 顶部使用，控制在 3 个以内：

```markdown
[![License](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)
[![Python](https://img.shields.io/badge/python-3.11+-green.svg)](requirements.txt)
```

禁止创建反映构建状态的徽章（如 CI/CD 状态），这属于内部信息。

## 七、链接规范

- 外部链接使用完整 URL
- 内部锚点使用相对路径
- 文档链接指向主要文档，而非子页面

## 八、语言风格

| 场景 | 推荐 | 避免 |
|------|------|------|
| 功能描述 | "支持 X" | "实现了 X 功能" |
| 操作说明 | "运行命令" | "您可以运行以下命令" |
| 状态描述 | "已完成" | "已经成功完成了" |
| 引用 | 使用反引号 | 使用引号 |

## 九、中英文混排

- 中文与英文之间加空格
- 英文与数字之间不加空格
- 示例：`Vue 3` `macOS` `100MB`

## 十、版本规范

- 使用语义化版本 (SemVer)：`v1.0.0`
- CHANGELOG 使用 Keep a Changelog 格式
- 版本间用空行分隔

## 十一、错误消息

- 用户可见的错误消息用中文
- 技术日志用英文
- 错误码格式：`ERR_XXX_001`

## 十二、交互式文档

- 使用 Details 块隐藏次要信息
- 使用 Tab 块展示多选项（如安装方式）
- 警告和注意使用方框样式

```markdown
> **注意** 这是一条提示

> **警告** 这是一条警告
```

## 违禁词汇

以下词汇禁止在文档中使用：

| 词汇 | 替代方案 |
|------|----------|
| "非常简单" | 直接描述 |
| "显而易见" | 直接描述 |
| "大家都用" | 直接描述 |
| "非常好" | 直接描述 |
| "只需" | 直接描述 |
| "轻松" | 直接描述 |

## 验收标准

1. 新用户能在 5 分钟内理解项目价值
2. 技术用户能在 10 分钟内完成第一个任务
3. 不需要任何外部文档即可上手
4. 目录结构清晰，不需要搜索
